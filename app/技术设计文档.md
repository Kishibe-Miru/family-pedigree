# 精神科遗传家族谱系图绘制工具 2.0 技术设计文档

## 1. 技术目标

本项目第一阶段目标是实现一个本地优先的网页工具，用于绘制精神科遗传家族谱系图，并导出为图片。

技术设计应满足：

- 可以本地运行。
- 可以部署到 Apache、Nginx、IIS 等局域网 Web 服务器。
- 不依赖外部网络。
- 不强制使用后端服务。
- 不强制使用数据库。
- 后续可以扩展为桌面应用。
- 后续可以接入后端，但第一阶段不需要。

## 2. 推荐技术路线

第一阶段采用纯前端静态网页方案。

核心技术：

- HTML：页面结构。
- CSS：界面样式。
- JavaScript：交互逻辑、数据模型、绘图、导出。
- SVG：绘制家族谱系图。
- Canvas：将 SVG 转换为 PNG 图片。
- JSON：保存和加载项目数据。
- localStorage：提供浏览器本地自动保存。

推荐结论：

> 使用纯前端静态网页实现，SVG 绘图，JSON 保存项目，Canvas 导出 PNG。

## 3. 技术方案分层

### 3.1 本地静态网页模式

运行方式：

- 双击 `index.html` 打开。
- 或使用本地静态服务器打开。

特点：

- 最简单。
- 不需要安装后端。
- 不需要数据库。
- 数据默认保存在用户本机。

限制：

- 浏览器直接打开时，部分文件 API 可能受到限制。
- 不适合多人共享同一份数据。

### 3.2 局域网静态部署模式

运行方式：

- 将项目目录放到 Apache、Nginx、IIS 或其他静态服务器。
- 局域网用户通过浏览器访问。

示例：

```text
http://192.168.1.20/family-pedigree/
```

特点：

- 多台电脑可访问同一个工具。
- 不需要外网。
- 不需要每台电脑复制程序文件。
- 每个用户仍可独立导入导出 JSON 项目文件。

限制：

- 如果没有后端，用户数据不会自动集中保存。
- 多人协同编辑同一张图需要后续后端支持。

### 3.3 桌面应用打包模式

后续可选方向：

- Electron
- Tauri

特点：

- 可以打包为 Windows 桌面软件。
- 文件读写体验更接近普通软件。
- 可支持最近文件、自动备份、默认保存目录等增强能力。

第一阶段不直接实现桌面打包，但代码结构应避免阻碍后续打包。

## 4. 暂不引入后端的原因

当前产品目标是绘制和导出图片，不涉及：

- 多人实时协作。
- 账号登录。
- 权限管理。
- 数据库检索。
- 服务器集中备份。
- 自动统计分析。

因此第一阶段不引入后端可以降低复杂度，提高开发速度。

只有出现以下需求时再考虑后端：

- 多人同时编辑同一份家系图。
- 科室或课题组统一管理病例家系图。
- 需要服务器集中备份。
- 需要账号和权限。
- 需要批量统计和检索。
- 需要生成分析报告。

## 6. 运行和部署要求

### 6.1 本地运行

推荐两种方式：

方式一：直接打开

```text
index.html
```

方式二：本地静态服务器

```text
http://localhost:端口号/
```

如果后续使用文件导入、导出、图片生成时出现浏览器限制，优先推荐使用本地静态服务器运行。

### 6.2 Apache 部署

将项目文件夹复制到 Apache 的 Web 根目录或站点目录。

示例：

```text
htdocs/family-pedigree/
```

访问：

```text
http://局域网IP/family-pedigree/
```

### 6.3 Nginx 部署

将项目目录配置为静态站点目录。

示例：

```nginx
location /family-pedigree/ {
    alias /path/to/family-pedigree/;
    index index.html;
}
```

### 6.4 IIS 部署

将项目目录添加为 IIS 静态网站或虚拟目录。

确保允许访问：

- `.html`
- `.css`
- `.js`
- `.json`
- 图片资源

## 7. 浏览器兼容性

优先支持现代桌面浏览器：

- Chrome
- Microsoft Edge
- Firefox

第一阶段不重点支持移动端。

最低要求：

- 支持 SVG。
- 支持 Canvas。
- 支持 Blob 下载。
- 支持 FileReader。
- 支持 localStorage。

## 8. 数据架构

项目数据使用 JSON 保存。

### 8.1 顶层数据结构

```json
{
  "version": "2.0",
  "title": "未命名家系图",
  "createdAt": "2026-05-26T00:00:00.000Z",
  "updatedAt": "2026-05-26T00:00:00.000Z",
  "people": [],
  "relationships": [],
  "settings": {
    "showNames": true,
    "showDiagnoses": true,
    "showLegend": true
  },
  "viewport": {
    "scale": 1,
    "offsetX": 0,
    "offsetY": 0
  }
}
```

### 8.2 人员数据结构

```json
{
  "id": "p1",
  "name": "II-3",
  "sex": "female",
  "age": "",
  "birthYear": "",
  "affectedStatus": "affected",
  "diagnoses": ["双相情感障碍"],
  "deceased": false,
  "proband": true,
  "notes": "",
  "x": 0,
  "y": 0,
  "manualPosition": false
}
```

字段说明：

- `id`：人员唯一 ID。
- `name`：显示名称或编号。
- `sex`：性别，支持 `male`、`female`、`unknown`。
- `age`：年龄。
- `birthYear`：出生年份。
- `affectedStatus`：疾病状态。
- `diagnoses`：诊断标签数组。
- `deceased`：是否死亡。
- `proband`：是否先证者。
- `notes`：备注。
- `x`、`y`：画布坐标。
- `manualPosition`：是否被用户手动调整过位置。

### 8.3 疾病状态

`affectedStatus` 可选值：

```text
unaffected
affected
suspected
unknown
```

含义：

- `unaffected`：未患病。
- `affected`：患病。
- `suspected`：疑似患病。
- `unknown`：未知。

### 8.4 关系数据结构

```json
{
  "id": "r1",
  "type": "partner",
  "person1": "p1",
  "person2": "p2",
  "status": "current"
}
```

关系类型：

```text
parentChild
partner
```

配偶关系状态：

```text
current
separated
divorced
unknown
```

亲子关系示例：

```json
{
  "id": "r2",
  "type": "parentChild",
  "parent": "p1",
  "child": "p3"
}
```

## 9. 状态管理设计

第一阶段使用单个全局状态对象管理项目数据。

示例：

```js
const state = {
  project: {},
  selectedPersonId: null,
  history: [],
  future: []
};
```

状态变更原则：

- 所有新增、删除、编辑操作都修改 `state.project`。
- 每次修改后重新渲染 SVG。
- 关键操作进入撤销历史。
- 保存时导出 `state.project`。
- 加载时替换 `state.project`。

## 10. 绘图设计

### 10.1 渲染方式

使用 SVG 绘制整个谱系图。

绘制内容包括：

- 人员节点。
- 配偶关系线。
- 亲子关系线。
- 先证者箭头。
- 死亡斜杠。
- 诊断文字。
- 图例。

### 10.2 节点符号

男性：

- SVG `rect`

女性：

- SVG `circle`

未知性别：

- SVG `polygon` 或旋转方形

患病：

- 节点填充深色。

疑似：

- 半填充或斜线纹理。

未知：

- 灰色或问号。

死亡：

- 在节点上绘制斜线。

先证者：

- 在节点旁绘制箭头指向节点。

### 10.3 图层顺序

建议按以下顺序绘制：

1. 背景。
2. 关系线。
3. 人员节点。
4. 状态符号。
5. 文字标签。
6. 选中高亮。
7. 图例和辅助控件。

关系线应位于节点下方，避免遮挡节点。

## 11. 自动布局设计

第一阶段采用规则布局，不引入复杂图布局引擎。

### 11.1 基础规则

- 父母位于子女上一代。
- 子女位于父母下一代。
- 配偶位于同一代。
- 兄弟姐妹位于同一水平线。
- 先证者尽量接近画布中心。

### 11.2 代际计算

可通过亲子关系计算 generation：

- 若存在先证者，以先证者为基准。
- 父母 generation = 子女 generation - 1。
- 子女 generation = 父母 generation + 1。
- 配偶 generation 尽量相同。

如果无法计算，放入默认代际。

### 11.3 手动位置

用户拖动节点后：

- 更新 `x`、`y`。
- 设置 `manualPosition = true`。

重新自动排版时可以提供两种模式：

- 保留手动位置。
- 全部重新排版。

第一阶段可以先实现全部重新排版。

## 12. 交互设计

### 12.1 选中人员

用户点击 SVG 节点后：

- 设置 `selectedPersonId`。
- 节点显示选中高亮。
- 右侧属性面板显示该人员信息。

### 12.2 添加成员

添加独立成员时：

- 创建新人员 ID。
- 使用默认性别 `unknown`。
- 使用默认疾病状态 `unknown`。
- 放置在当前画布中心或默认位置。

### 12.3 添加关系成员

选中人员后，可添加：

- 父亲
- 母亲
- 配偶
- 子女
- 兄弟姐妹

添加时同时创建成员和关系。

### 12.4 删除成员

删除成员时：

- 删除该人员。
- 删除所有与该人员相关的关系。
- 清空选中状态。
- 重新渲染。

### 12.5 拖拽节点

拖拽节点时：

- 鼠标按下记录起点。
- 鼠标移动更新坐标。
- 鼠标释放写入最终位置。
- 重新绘制关系线。

### 12.6 缩放和平移

画布支持：

- 鼠标滚轮缩放。
- 按住空白区域拖动画布。
- 重置视图。

第一阶段如时间有限，可先实现节点拖动，缩放和平移后续补充。

## 13. 保存和加载设计

### 13.1 自动保存

使用 `localStorage` 保存最近一次项目数据。

键名建议：

```text
psychiatric-pedigree-v2-autosave
```

每次关键变更后自动保存。

### 13.2 手动保存

用户点击保存时：

- 将 `state.project` 转为 JSON。
- 创建 Blob。
- 触发浏览器下载。

文件名建议：

```text
家系图_YYYYMMDD_HHmm.json
```

### 13.3 手动加载

用户选择 JSON 文件后：

- 使用 FileReader 读取。
- 校验 JSON 格式。
- 检查 `version` 字段。
- 加载到 `state.project`。
- 重新渲染。

### 13.4 兼容性处理

如未来数据结构升级，应提供基础迁移逻辑。

第一阶段仅支持 `version = 2.0`。

## 14. 图片导出设计

### 14.1 PNG 导出流程

1. 获取当前 SVG 内容。
2. 克隆 SVG，补齐样式。
3. 序列化为字符串。
4. 转为 Blob URL。
5. 创建 Image 对象。
6. 将 Image 绘制到 Canvas。
7. 从 Canvas 生成 PNG。
8. 触发下载。

### 14.2 导出边界

导出时应计算图形实际边界，避免：

- 大量空白。
- 节点被裁切。
- 文字被裁切。

第一阶段可先导出完整 SVG 视口，后续优化为按内容边界导出。

### 14.3 导出尺寸

建议默认导出 2 倍像素密度，提升清晰度。

例如：

- 画布显示尺寸：1200 x 800
- 导出尺寸：2400 x 1600

## 15. 撤销和重做

第一阶段建议预留结构。

可使用：

```js
history: []
future: []
```

每次关键操作前保存一份项目快照。

支持操作：

- 撤销
- 重做

如果 MVP 时间有限，可以先做数据结构预留，后续实现按钮功能。

## 16. 错误处理

需要处理以下错误：

- 加载的 JSON 格式错误。
- 加载的文件不是项目文件。
- 删除成员后关系异常。
- 图片导出失败。
- 浏览器不支持某些 API。

错误提示应显示在页面中，不只使用控制台。

## 17. 性能考虑

第一阶段目标家系规模：

- 20 到 100 名成员。

性能策略：

- 每次状态变化后重绘整个 SVG。
- 避免复杂动画。
- 避免外部依赖。

如果后续支持更大家系，再考虑局部渲染和布局优化。

## 18. 安全和隐私

项目涉及家族史和精神科诊断信息，应默认本地保存。

第一阶段要求：

- 不上传任何数据。
- 不调用外部接口。
- 不使用外部 CDN。
- 不嵌入第三方统计。
- 不自动联网。

用户手动导出的 JSON 和 PNG 由用户自行保存和管理。

可扩展方向包括：

- SVG 导出。
